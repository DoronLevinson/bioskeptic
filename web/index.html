MCP & Python Library

BioSkeptic's engine isn't locked inside the chat. The same grounded checks are available two other ways — as an MCP server, so any AI agent can call them as tools, and as a Python library for batch data refinement. Both are free and local: they run on your machine with your own API key, and nothing is sent to a BioSkeptic server.

The MCP server — BioSkeptic as tools for any agent

MCP (Model Context Protocol) is the standard that lets an AI agent call external tools. BioSkeptic's MCP server exposes its whole toolbox, so instead of you asking BioSkeptic, your agent — Claude Desktop, Claude Code, or any MCP client — can reach for these tools mid-reasoning and get back grounded, cited results.

The tools it provides:

  • Resolvers (resolve_target, resolve_drug, resolve_disease) — turn a messy name, or a knowledge-graph id like Gene::7157, into a grounded ID passport the other tools can use.
  • build_report — the core. Runs every refuting mechanism on a drug–target–disease claim and returns a structured report: the concerns that fired, the checks that passed, each mechanism's precision, and cited links.
  • Dig tools (search_trials, search_pubmed, fda_label) — chase a concern down to primary evidence (clinical trials, papers, the FDA label), keyed on the ids already resolved.
  • drug_targets — a drug's direct mechanism-of-action target(s), for when a claim names a drug but not a target.

To use it, add BioSkeptic to your Claude Desktop config (Settings → Developer → Edit Config) and restart — then just ask it to red-team a claim:

{
  "mcpServers": {
    "bioskeptic": {
      "command": "uv",
      "args": ["run", "python", "-m", "mcp_server.server"],
      "cwd": "/path/to/bioskeptic",
      "env": { "ANTHROPIC_API_KEY": "sk-ant-..." }
    }
  }
}

The library — bioskeptic.refine

For pipelines rather than chat. bioskeptic.refine runs the same refuting panel over the target–disease relations in a dataset or knowledge graph and flags the weakly-grounded ones — each with a cited reason — so you can clean a dataset, refine a KG, or train repurposing models on relations that actually hold up.

Audit a single relation:

from bioskeptic.refine import audit_pair

r = audit_pair(target="TP53", disease="type 2 diabetes")
r.flagged          # True
print(r.details)   # reasons + source links
r.findings         # structured: [{mechanism, title, reason, sources}, ...]

…or a whole dataframe — it adds flagged, flags, flag_mechanisms, and findings columns:

from bioskeptic.refine import audit_dataframe
out = audit_dataframe(df)      # df with 'target' and 'disease' columns
out[out["flags"] >= 2]         # keep only corroborated flags

The Data Refinement page shows a worked example on 100 relations sampled from DRKG.

Free & local

Both run on your machine with your own key. Five of the eight mechanisms — and all entity resolution — are free public-database lookups (Open Targets, GTEx, the Human Protein Atlas, IMPC, and friends). The other three call an LLM to judge relevance, costing roughly a few cents per audit on your key. There's nothing hosted to rate-limit or trust — you run it, you own it.

Data Refinement

BioSkeptic isn't only a chat assistant. The same grounded engine is available as an API that runs its panel of refuting mechanisms over the edges of a dataset or knowledge graph and flags the relations that don't hold up — each with a cited, human-readable reason.

This matters because the datasets and knowledge graphs behind modern drug discovery are noisy. A large share of their target–disease links come from text-mining papers that merely co-mention two entities — a connection that can exist with no real biology behind it. Train a repurposing model on those edges and you teach it to trust connections that aren't there.

A worked example — auditing DRKG

To show what this looks like, we took DRKG (the Drug Repurposing Knowledge Graph, a widely used biomedical KG) and randomly sampled 100 target–disease relations. Each one runs through the API, which resolves the gene and disease and then checks them against every refuting mechanism — is the target actually expressed in the affected tissue? does deleting it in mice do anything relevant? is there a genetic signal, or only paper co-mention?

We flagged a relation whenever 3 or more independent mechanisms fired — a deliberately strict bar, so a flag means several unrelated lines of evidence all point the same way.

The results

16 of the 100 relations were flagged. On manual review, 13 were genuine — targets with no real footprint in the disease, mis-categorised genes, or edges resting only on literature co-mention — a precision of ~81%, at the strictest confidence tier.

A concrete catch: DRKG links EPO (erythropoietin) to inherited thrombocytopenia — but EPO drives red blood cell production, not platelets (that's thrombopoietin). Sure enough, it isn't expressed in blood, its mouse knockout causes anemia rather than a platelet defect, and there's no genetic support — and every one of those reasons comes with a database link. The edge is almost certainly a lineage mix-up.

The few false positives shared a single, predictable signature: secreted growth factors and structural proteins (like FGF2) that are genuinely disease-relevant but trip the single-cell check because they act between cells rather than inside the disease-driving one. Because they cluster, they're easy to filter — and precision climbs further if you require a mouse-knockout or genetic mechanism rather than expression alone.

Why this matters at scale

One hundred edges is a demo; the value is in the millions. Run at scale, the API can:

  • Refine training datasets — strip the weakly-grounded edges before they teach a model bad priors.
  • Clean knowledge graphs — surface candidate spurious links for curators, each with a cited rationale instead of a bare score.
  • Sharpen drug-discovery & repurposing algorithms — models trained and evaluated on grounded, audited relations make better, more trustworthy predictions.

Every flag is backed by a source you can open and check — so this isn't a black-box filter, it's an auditable second opinion on every connection in your data.
